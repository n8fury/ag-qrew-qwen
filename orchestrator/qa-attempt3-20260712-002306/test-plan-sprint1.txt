Change type:     New
Risk level:      Medium
Affected areas:  Sign-in page, REST API /api/auth/login
Dependencies:    None

SFDIPOT Coverage Map
  Structure:   Sign-in form, error messages, success message, redirect to /tasks
  Function:    Submit valid credentials, submit invalid credentials, handle session persistence
  Data:        Email (valid format), password (min length, max length)
  Interfaces:  POST /api/auth/login (success, failure)
  Platform:    Desktop, mobile
  Operations:  Session persistence, rate limiting, concurrent logins
  Time:        Timeout mid-login, date/time edges not applicable

Oracles (FEW HICCUPPS)
  Claims - behaviour stated in the source doc / spec
  History - consistent with how this flow behaved before the change
  User expectations - what a reasonable user would expect here
  Purpose - serves the stated business intent

What MUST work (positive expected results)
  Scenario                              | Expected Result
  --------------------------------------|----------------------------------------
  Valid user signs in with correct credentials | Redirect to /tasks, session cookie set
  Admin user signs in with correct credentials | Redirect to /tasks, session cookie set, admin privileges granted

What MUST be handled (negative expected results)
  Scenario                              | Expected Result
  --------------------------------------|----------------------------------------
  Invalid email format                 | Error message displayed, no redirect
  Incorrect password                   | Error message displayed, no redirect
  Missing email or password            | Error message displayed, no redirect
  Rate limit exceeded                  | Error message displayed, no redirect
  Unauthorized access attempt          | Redirect to /, session cookie not set

  - None identified

--- Feature: Task Management (CRUD) ---

Change type:     New
Risk level:      Medium
Affected areas:  Tasks page, REST API /api/tasks
Dependencies:    Authenticated session

SFDIPOT Coverage Map
  Structure:   Task list, task form, task details, delete confirmation, count display
  Function:    List tasks, create task, update task, delete task, validate input
  Data:        Task title (required, max 200 characters), done flag (boolean)
  Interfaces:  GET /api/tasks, POST /api/tasks, PUT /api/tasks/:id, DELETE /api/tasks/:id
  Platform:    Desktop, mobile
  Operations:  Session persistence, rate limiting, concurrent operations
  Time:        Timeout mid-operation, date/time edges not applicable

Oracles (FEW HICCUPPS)
  Claims - behaviour stated in the source doc / spec
  Comparable - consistent with sibling features in the same product
  User expectations - what a reasonable user would expect here
  Purpose - serves the stated business intent

What MUST work (positive expected results)
  Scenario                              | Expected Result
  --------------------------------------|----------------------------------------
  List tasks with valid session         | Current list of tasks shown, accurate count
  Create task with valid title          | New task added to list, count updated
  Update task with valid title          | Task details updated, count unchanged
  Delete task with valid ID             | Task removed from list, count updated
  Toggle task done flag                 | Task status updated, count unchanged

What MUST be handled (negative expected results)
  Scenario                              | Expected Result
  --------------------------------------|----------------------------------------
  Missing task title                    | 400 error, task not created, count unchanged
  Over-length task title                | 400 error, task not created, count unchanged
  Unauthenticated request               | 401 error, no access to tasks, count unchanged
  Non-existent task ID                  | 404 error, no task deleted, count unchanged
  Concurrent modification               | Conflict error, task not updated, count unchanged
